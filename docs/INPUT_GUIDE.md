# 输入说明

这套系统当前把输入分成两类：

1. 动态源文件
   - 放在固定目录 `源文件/`
   - 系统会自动扫描、识别和选择本次运行实际使用的文件
2. 固定配置文件
   - 放在 `config/`
   - 系统按固定文件名读取，不参与“最新文件”选择

正式输入契约配置文件：
- `/Users/ahs/Desktop/Operations Analytics Engine/config/input_sources.json`

## 1. 默认输入目录

### 动态源文件目录
- `源文件/`

当前主链会从这里扫描：
- 直播进度表
- 线索明细
- 成交明细

### 固定配置目录
- `config/`

当前主链固定读取：
- `config/monthly_targets.csv`
- `config/daily_spend.csv`
- `config/manual_attribution_overrides.csv`

## 2. 命名规范

### 直播进度表
- 正式命名：`2026年直播进度表3月.xlsx`
- 允许格式：`.xlsx` / `.xls`
- 判定逻辑：按文件名中的业务月份识别最新文件

### 线索明细
- 正式命名：`总部新媒体线索2026-03-17.csv`
- 允许格式：`.csv` / `.xlsx` / `.xls`
- 判定逻辑：按文件名中的业务日期识别最新文件

### 成交明细
- 正式命名：`总部新媒体成交2026-03-17.csv`
- 允许格式：`.csv` / `.xlsx` / `.xls`
- 判定逻辑：按文件名中的业务日期识别最新文件

### 固定配置文件
- `monthly_targets.csv`
- `daily_spend.csv`
- `manual_attribution_overrides.csv`

这两类文件不参与“最新文件”选择，只认固定路径。

## 3. 最低必要字段

### 直播进度表
最低必要字段：
- 日期
- 开播账号
- 本场主播
- 全场景线索人数

可选字段：
- 开播时间
- 下播时间
- 消耗

说明：
- 缺少开播/下播时间时，事实层会退回全天窗口匹配，但系统会给出警告
- 缺少 `全场景线索人数` 会直接拦截，因为日报线索质量口径要用

### 线索明细
最低必要字段：
- 线索ID
- 创建时间
- 渠道2
- 手机号
- 渠道3

说明：
- 这几个字段缺任何一个，都会在主链开始前拦截
- 别名字段也支持，例如 `ID`、`线索创建时间`、`三级渠道`

### 成交明细
最低必要字段：
- 线索ID
- 订单状态

可选字段：
- 成交时间
- 账号
- 渠道
- 成交车型
- 手机号

### 月目标配置
最低必要字段：
- month
- scope_type
- scope_name
- parent_account
- lead_target_month
- deal_target_month

### 消耗配置
最低必要字段：
- date
- account
- actual_spend

### 专项人工确认归属
最低必要字段：
- override_id
- business_subject_key / phone / lead_id（至少能定位样本）
- override_scope
- target_account / target_host（至少填一个）
- reason
- evidence_note
- confirmed_by
- confirmed_at
- status

说明：
- 这是覆盖层，不会改写自动归因事实表
- 这份文件会影响日报、分析、导出等最终消费口径
- 具体填写规则请看 `docs/MANUAL_ATTRIBUTION_GUIDE.md`

## 4. 最新文件判定逻辑

### 动态源文件
系统会先扫描目录下所有符合 glob 规则的候选文件，再做两层判断：

1. 命名是否符合正式规范
2. 文件名里能不能识别业务日期 / 业务月份

然后按下面规则挑选本次实际使用文件：

- 直播进度表：按业务月份选最新
- 线索明细：按业务日期选最新
- 成交明细：按业务日期选最新

### 冲突处理
如果出现以下情况，系统会在最前面直接中止：
- 找不到任何候选文件
- 扫描到了文件，但命名都不符合规范
- 同一业务日期存在多个候选，无法安全判断
- 字段不完整

系统不会静默选一个“看起来差不多”的文件继续跑。

## 5. 本次到底用了哪些输入，去哪里看

每次主链运行，都会额外生成一份输入清单：

- `artifacts/runs/input_manifest_{run_id}.json`

里面会记录：
- 文件类别
- 文件路径
- 文件名
- 最后修改时间
- 业务日期
- 识别依据
- 字段校验是否通过
- 本次运行编号

同样的信息也会写入：
- `artifacts/runs/run_manifest_{run_id}.json`
- `artifacts/runs/quality_report_{run_id}.json`

## 6. 常见输入错误和处理方式

### 情况1：缺少直播进度表
表现：
- 系统在主链开头直接报错：未找到直播进度表

处理：
- 把当月直播进度表放进 `源文件/`
- 文件名按 `2026年直播进度表3月.xlsx` 这种形式命名

### 情况2：线索文件放进去了，但命名不规范
表现：
- 系统提示“扫描到文件，但命名均不符合规范”

处理：
- 改成 `总部新媒体线索YYYY-MM-DD.csv`

### 情况3：同一天有两个线索文件
表现：
- 系统提示“存在多个同业务日期候选，无法安全判定”

处理：
- 只保留一个正式文件
- 或明确指定路径

### 情况4：字段名变了
表现：
- 系统在开始前就报“字段校验失败”

处理：
- 对照输入契约恢复字段名
- 或补齐当前别名规则后再跑

### 情况5：日报里“原始线索全场景人数”是 0
表现：
- 通常说明直播进度表没选对，或者缺少 `全场景线索人数`

处理：
- 先检查 `input_manifest`
- 再检查直播进度表字段

## 7. 最简单的使用方式

如果你只跑正式主链，直接运行：

```bash
python -m oae.jobs.daily_pipeline
```

系统会自动：
- 扫描 `源文件/`
- 选中本次实际使用的直播/线索/成交文件
- 校验字段
- 生成 `input_manifest`
- 再进入 fact -> snapshot -> ledger -> analysis -> export -> quality

如果你只想单独跑事实层：

```bash
python -m oae.cli.build_fact
```

默认也会优先从 `源文件/` 自动找线索和成交文件，直播进度表留空时自动选最新。
