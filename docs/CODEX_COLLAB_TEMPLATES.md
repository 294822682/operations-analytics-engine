# 我与 Codex 的高频协作模板

这份文档只解决一个问题：

**以后你怎么一句话把任务交给 Codex，并稳定拿到你要的成品。**

不要把它当工程文档。

把它当成：

- 你的固定指令库
- 你的高频工作模板
- 你和 Codex 之间的标准协作语言

---

## 1. 最重要的使用原则

以后和 Codex 协作，尽量遵守这 5 条：

### 1.1 先说你要的成品，不要先讲一大堆背景

错误方式：

“我最近在想我们的线索经营好像哪里不太对，你帮我看看……”

更好的方式：

“基于最新全链路结果，给我一份后链路复盘汇报，面向老板，纯文本输出。”

也就是说：

- 先说成品是什么
- 再说面向谁
- 最后说格式要求

### 1.2 尽量说“让我直接产出”，不要说“帮我分析一下”

“分析一下”容易得到中间过程。

“直接生成日报文本 / 复盘汇报 / 口播稿 / TSV”更容易得到可直接使用的结果。

### 1.3 尽量指定受众

同一份数据，面对不同对象，写法完全不同。

你最好明确写：

- 面向老板
- 面向经营负责人
- 面向运营负责人
- 面向内部执行同事
- 面向自己看

### 1.4 尽量指定输出形式

常见输出形式只有这几种：

- 纯文本汇报
- TSV
- Markdown
- 口播稿
- 简短结论
- 排查清单

### 1.5 默认让 Codex 先跑正式链路，再给结果

如果你要的是最新结果，尽量直接说：

“先跑全链路，再输出……”

这样就不会拿旧产物说事。

---

## 2. 你最常用的 4 类任务

以后你和 Codex 的高频协作，基本就围绕这 4 类：

1. 日报
2. 绩效
3. 复盘
4. 异常排查

你只要把这 4 类模板用熟，效率就会明显高很多。

---

## 3. 日报模板

### 3.1 标准版

适合每天直接让 Codex 出日报。

```text
先跑全链路，然后基于最新结果生成一份日报。

要求：
1. 面向管理层
2. 先给文本版日报
3. 再给一份 TSV，方便我直接粘贴到 Excel
4. 必须包含：
   - 部门总体
   - 成交账号
   - 线索质量口径
   - 账号层
   - 到人层
5. 如果质量校验没通过，要先告诉我，不要直接装作正常日报输出
```

### 3.2 极简版

```text
跑今天日报，给我文本版 + TSV版。
```

### 3.3 领导汇报版

```text
先跑最新日报链路，然后给我一份适合领导直接看的日报文本。

要求：
1. 不要像表格抄录
2. 先说 3 条最重要结论
3. 再给关键数字
4. 最后补 2-3 条经营动作
```

### 3.4 只要 Excel 粘贴版

```text
先跑日报链路，然后只输出 TSV。

要求：
1. 我可以直接复制到 Excel
2. 保持“日报日期 / 部门总体 / 成交账号 / 线索质量口径 / 账号层 / 到人层”结构
3. 不要额外解释
```

## 4. 绩效模板

### 4.1 标准版

```text
先跑 fact -> snapshot -> ledger，然后给我当前最新绩效口径结果。

要求：
1. 以 CompensationLedger 为准
2. 不要拿日报展示表替代绩效台账
3. 输出内容至少包含：
   - 结算周期
   - 账号/主播范围
   - leads / deals / spend
   - CPL / CPS
   - 目标达成率
4. 如果有冻结编号或 freeze_id，要明确写出来
```

### 4.2 简版

```text
跑绩效链，按最新台账给我结算口径结果。
```

### 4.3 绩效异常排查版

```text
基于最新 CompensationLedger 和 DailyPerformanceSnapshot，
帮我找出最值得复核的 3 个绩效异常。

要求：
1. 说清楚异常点是什么
2. 说清楚更像口径问题还是经营问题
3. 给出排查优先级
```

## 5. 后链路复盘模板

这是你最常用、也最值钱的一类。

### 5.1 标准复盘版

```text
先跑全链路，然后基于最新 analysis 结果，
生成一份《线索经营后链路复盘汇报》。

要求：
1. 纯文本输出，不要 HTML
2. 面向老板 / 经营负责人
3. 不要逐表解释
4. 要写成经营复盘汇报，不要写成技术摘要
5. 至少包含：
   - 核心结论
   - 主漏斗诊断
   - 成交双口径分析
   - SLA 时效分析
   - 渠道与账号表现
   - 主播表现
   - 车型路径
   - 区域潜力
   - 回收链路
   - 健康度
   - 数据质量
   - 管理动作建议
6. 每个模块都要有结论，不要只报数
```

### 5.2 老板汇报版

```text
基于最新 analysis 结果，
给我一份适合老板听的线索经营后链路复盘汇报。

要求：
1. 纯文本
2. 先说结论，再补数字
3. 不讲技术实现
4. 强调：
   - 哪里的问题最大
   - 哪里的风险最高
   - 哪些结果不能误判
   - 下一步该做什么
```

### 5.3 口播稿版

```text
基于最新 analysis 结果，
给我一份 3 分钟内能讲完的经营复盘口播稿。

要求：
1. 纯文本
2. 语言像人在汇报，不像分析报告
3. 每段先说结论
4. 最后给 3 条管理动作
```

### 5.4 只要问题和动作

```text
基于最新后链路分析，不要大篇幅复述数据。

直接告诉我：
1. 当前最关键的 3 个经营问题
2. 每个问题背后的原因判断
3. 如果本周只能做 3 件事，最该做什么
```

## 6. 异常排查模板

### 6.1 通用异常排查

```text
先跑全链路，然后帮我找今天最值得排查的 3 个异常。

要求：
1. 结合 fact / snapshot / analysis / quality 一起判断
2. 每个异常要写：
   - 现象
   - 可能原因
   - 更像数据问题还是经营问题
   - 排查优先级
3. 不要泛泛而谈
```

### 6.2 无主异常

```text
基于最新 fact 和 analysis，
帮我判断今天无主线索为什么高。

要求：
1. 先说明无主比例和主要集中账号
2. 再判断更像直播进度表问题、账号映射问题、还是归因规则问题
3. 最后给出最先要查的 3 个点
```

### 6.3 成交异常

```text
基于最新日报和 analysis，
帮我解释为什么今天线索不差，但成交偏弱。

要求：
1. 不要只复述数字
2. 要从漏斗、SLA、账号、主播、沉淀转化几个角度判断
3. 直接给经营解释
```

## 7. 周报 / 周复盘模板

### 7.1 周总结

```text
基于最近 7 天全链路结果，
生成一份经营周总结。

要求：
1. 面向经营负责人
2. 先说本周最重要的 3 个变化
3. 再说本周最值得复盘的 3 个问题
4. 最后给出下周最该做的 3 个动作
```

### 7.2 周异常汇总

```text
把最近 7 天最重要的经营异常汇总给我。

要求：
1. 不要按天流水账
2. 要按问题类型归纳
3. 每类问题都要有结论和动作建议
```

## 8. 给别的部门复用时怎么问

当数据口径不变，只变化账号、主播、目标时，你可以直接这样让 Codex干活。

### 8.1 迁移评估

```text
请基于当前产品化后的主链，
评估这套系统是否可以复用到另一个部门。

前提：
1. 经营口径不变
2. 主体模型不变
3. 只变化账号、主播、目标、部分配置

请直接告诉我：
1. 哪些配置要换
2. 哪些模块不用改
3. 哪些地方最容易踩坑
4. 怎么做最稳妥的接入
```

### 8.2 直接生成接入清单

```text
请按“另一个部门只换账号和主播，经营口径不变”的前提，
给我一份最小接入清单。

要求：
1. 用中文
2. 写给业务 owner 看
3. 明确区分：
   - 必须改的配置
   - 不要动的事实层逻辑
   - 需要重新验证的输出
```

### 8.3 迁移后跑一版

```text
请按新的账号和主播配置，跑一版全链路。

要求：
1. 先检查配置是否完整
2. 再跑事实层、快照、台账、分析
3. 最后告诉我：
   - 是否跑通
   - 哪些产物已经生成
   - 哪些问题需要修
```

## 9. 什么时候应该让 Codex“直接跑”，什么时候只让它“解释”

### 9.1 应该直接跑

遇到这些情况，直接让 Codex 先跑：

- 你要最新日报
- 你要最新绩效
- 你要最新后链路复盘
- 你怀疑今天数据异常
- 你要确认主链有没有坏

关键词可以直接写：

- “先跑全链路”
- “按最新结果”
- “基于当前最新产物”

### 9.2 先不要跑，只解释

遇到这些情况，可以先问解释：

- 这几个对象分别是什么
- 这个指标口径怎么算
- 这条链路为什么这么设计
- 如果接入别的部门，需要改什么

## 10. 最容易成功的提问公式

以后你可以尽量按这个顺序说：

```text
先跑哪条链路
-> 要什么成品
-> 面向谁
-> 用什么格式
-> 有什么特别要求
```

比如：

```text
先跑全链路，
基于最新结果生成一份线索经营后链路复盘汇报，
面向老板，
纯文本输出，
不要逐表解释，要先讲结论，再讲问题，最后给动作建议。
```

这就是一个非常标准、成功率很高的指令。

## 12. 你以后最该养成的习惯

以后你和 Codex 协作，最好的工作方式是：

### 第一，不要让自己做中间加工

不要自己先拼数字，再让我总结。  
直接让我基于正式产物输出你要的成品。

### 第二，不要每次重新解释系统

系统结构已经固定了。  
以后尽量直接说你要什么结果，不要每次重新讲一遍背景。

### 第三，把高频需求模板化

你以后真正高频的，基本就是：

- 日报
- 绩效
- 复盘
- 异常
- 动作建议

把这些模板用熟，效率会比“每次重新想怎么问”高很多。

---

## 13. 一句话总结

你以后最有效的用法，不是“让我帮你想”，而是：

**直接让我跑正式链路，然后按你指定的受众和格式，把结果变成成品。**
